# code review

## 范围
对 `dayu/` 下所有代码进行 code review。

## 补充约束（避免误报）
1. 工具注册中的嵌套函数：若用于装饰器注册 + 闭包捕获 service/limits/registry，默认视为合理；仅在闭包状态复杂、难测或重复逻辑明显时提出重构建议。
2. 工具 schema必须把字符串写在schema内，禁止使用常量定义，工具schema使用magic string默认视为合理。

## review结果
- 写入 workspace/code-review-mmdd-HHMM.md （月份日期-小时分钟）。
- 每发现一项编号后写入一项。
- review结束给出按严重程度排序等完整列表。

## 检查项

### 0. 业务逻辑

#### 目标

- 本节只检查一件事：代码在真实执行链上的“输入 -> 分支 -> 输出 / 副作用”是否正确，而不是只看实现风格是否“像是对的”。
- 业务逻辑 review 分两条主线同时进行：
  - **端到端执行链**：从 `CLI / WeChat` 入口接收用户输入、命令行参数、prompts、config/override 开始，到 answer / error / interrupt / resume 结果回到 UI 结束。
  - **函数级执行链**：从函数入参开始，到分支条件、下游调用、返回值、异常、副作用结束。
- root cause 必须基于**同一条代码路径上的直接证据**定位：输入从哪里进入、在哪个条件分支被判断、最后产生了什么返回值/状态变化。禁止仅凭现象、命名或局部实现风格做猜测性归因。

#### 0.1 端到端执行链路是否闭环正确

- 以真实入口为起点检查完整主链路：`UI -> Service -> Contract preparation -> Host -> scene preparation -> Agent -> Host -> UI`。
- 对 `CLI`、`WeChat` 等每个入口，至少检查这些分支是否都能沿链路自洽推进：
  - 正常输入 -> 正常 answer 返回 UI。
  - 正常输入 + tool 调用 -> tool 结果回填 -> 继续执行 / 结束。
  - tool 失败 / model 失败 / timeout / cancel / interrupt -> 是否进入预期失败分支。
  - resume / retry / redelivery 分支 -> 是否基于稳定快照恢复，而不是依赖当前环境临时反推。
  - 非法输入 / 非法 override / 缺失 prompt / 缺失 config / 不支持的参数组合 -> 是否被明确拒绝、降级或兜底处理。
- 判定标准：任一执行分支若存在“参数已传入但未生效”“分支应触发却未触发”“错误被错误吞掉继续执行”“结果返回 UI 时语义已漂移”，即报告。

#### 0.2 参数传递、覆盖优先级与生效点是否正确

- 检查从入口传入的参数是否按预期一路传递到真正消费它的层，包括但不限于：
  - CLI 参数 / WeChat 请求参数
  - workspace/config 默认配置
  - prompt asserts / prompts 内容
  - execution options / overrides
  - Host / Agent 最终实际读取的运行参数
- 对每个关键参数，检查四件事：
  - **来源**：参数从哪里进入系统。
  - **覆盖关系**：默认值、配置值、请求值、override 的优先级是否一致。
  - **落点**：参数最终在哪一层、哪个函数、哪个字段被消费。
  - **失效处理**：非法值、空值、缺失值、类型错误值是否被明确处理，而不是静默忽略。
- 重点检查：
  - 参数是否只传到了中间对象但没有被最终执行层读取。
  - 参数是否在链路中被重新默认化、覆盖、丢失或被不同层以不一致方式解释。
  - 非预期参数是否错误落入“正常分支”继续执行，造成语义漂移。
- 判定标准：只要“传入值”和“最终生效值”之间无法建立直接、可追踪的数据链，或优先级实现与调用方预期不一致，即报告。

#### 0.3 分支条件与路由是否基于正确事实

- 检查所有关键 if/elif/match/dispatch 路由是否由**真正决定该分支的事实**驱动，而不是由间接信号、历史遗留标记或碰巧相关的字段驱动。
- 检查分支顺序是否正确：
  - 是否存在宽条件抢先命中，导致更具体分支永远进不去。
  - 是否存在条件重叠但没有明确优先级，导致行为依赖书写顺序。
  - 是否存在遗漏 `else` / 默认分支，导致异常输入悄悄落空。
- 对 dispatch / router / builder / handler，检查是否同时承担了“参数转换 + 分支选择 + 结果合并”过多职责，从而让某些分支逻辑被错误复用或漏判。
- 判定标准：分支条件若不能直接表达真实业务含义，或不同输入命中错误分支 / 落不到任何分支，即报告。

#### 0.4 函数级 review：入参 -> 分支 -> 返回值 / 异常

- 对关键 public 函数，以及分支复杂、状态更新复杂、I/O 边界明显的 private helper，逐个检查：
  - **入参域**：函数预期接受哪些正常输入。
  - **异常入参**：`None`、空字符串、空列表、缺失字段、非法枚举、错误类型、越界值、互相冲突的参数组合是否被处理。
  - **分支逻辑**：不同输入是否进入预期分支，是否存在不可达分支、漏判分支、死分支。
  - **下游调用假设**：调用下游前是否已经满足下游前置条件。
  - **返回契约**：返回值、异常、状态变更是否与函数名、docstring、调用方预期一致。
- 每个重要函数都要按这个顺序看清楚：`入参 -> 条件判断 -> 下游调用 -> 返回值 / raise -> 副作用`。
- 重点报告以下问题：
  - 入参校验缺失，导致非法值深入下游才崩。
  - 某分支更新了部分状态后抛错，留下半提交状态。
  - 返回类型或返回语义在不同分支不一致，调用方无法稳定消费。
  - 实际会返回 `None` / 空对象 / 半成品对象，但调用方按“成功结果”继续推进。

#### 0.5 非预期输入是否被稳定处理

- review 时不要只验证 happy path，必须主动寻找 non-happy path：
  - 缺失必填参数
  - 类型错误
  - 非法枚举 / 非法字符串
  - 空内容 / 空消息 / 空 prompt
  - 越界数值 / 负数 / 超长输入
  - 重复请求 / 冲突参数 / 已结束状态再次推进
- 检查面对非预期输入时，代码是否采取了**明确且稳定**的策略之一：
  - 拒绝并返回清晰错误
  - 合法归一化
  - 安全降级
  - 有边界的重试
  - 安全 no-op
- 判定标准：若代码对非预期输入只是“碰巧没崩”“吞错继续跑”“落入正常分支但语义已错”，即报告。

#### 0.6 返回值、副作用与外部可见结果是否一致

- 业务逻辑 review 不只看返回文本；还要同时检查：
  - 返回给上层 / UI 的结果
  - 持久化状态更新
  - transcript / pending turn / resume snapshot / reply outbox 等外部状态
  - event / log / trace 中记录的执行事实
- 检查是否存在这些不一致：
  - 函数返回“成功”，但实际状态未提交或提交了一半。
  - UI 看到“已完成”，但 Host / 存储侧仍处于可恢复或运行中状态。
  - 错误分支返回了看似成功的默认值，掩盖真实失败。
  - 上层依据返回值认为某参数已生效，但下游实际没有按该参数执行。
- 判定标准：同一执行事实在“返回值 / 持久化状态 / 外部可见状态”中若出现矛盾，即报告。

#### 0.7 产出要求

- 每条业务逻辑问题至少写清楚以下内容：
  - **入口/函数**：问题发生在什么执行入口或什么函数。
  - **输入场景**：什么输入会触发问题，区分预期输入还是非预期输入。
  - **实际分支**：代码实际走到了哪个分支。
  - **预期行为**：按当前系统设计，应该如何处理。
  - **实际行为**：现在返回了什么、写入了什么状态、或漏做了什么。
  - **直接证据**：指出具体判断条件、参数传递路径、返回值或状态更新位置。
  - **影响**：会导致错误 answer、错误状态、静默失效、不可恢复，还是仅局部行为错误。
- 禁止只写“参数可能没传下去”“这里可能分支有问题”“这里看起来不太对”。必须把问题收敛到可复现的输入、可定位的分支、可证明的错误结果。


### 1. 架构与依赖

#### 约束（检查时参照，不需要在产出中复述）

四层架构 `UI -> Service -> Host -> Agent`，层到代码目录的映射：

| 层 / 模块 | 代码目录 | 职责边界 |
|---|---|---|
| **UI** | `cli/`, `gui/`, `web/`, `wechat/` | 用户交互入口 |
| startup preparation（UI 的 public 模块） | `startup/` | 启动期把原始路径收敛成稳定依赖，不参与请求期控制流 |
| **Service** | `services/` | 唯一业务理解层 |
| Contract preparation（Service 的 public 模块） | `services/contract_preparation.py` | 机械装配 Execution Contract |
| **Host** | `host/` | 通用托管执行层，无业务理解 |
| scene preparation（Host 的 public 模块） | `host/scene_preparer.py`, `prompting/` | 按 scene 策略机械收敛执行约束 |
| **Agent** | `engine/` | 执行已准备好的消息交互，无业务理解 |
| 跨层数据契约 | `contracts/`, `execution/` | 纯数据定义，不含行为 |
| 领域能力 | `fins/` | 金融工具和处理能力 |

禁止反向依赖：下层不得 import 上层。合法方向：UI → Service → Host → Agent。

#### 检查项（逐项执行）

**1.1 反向依赖检测**
- 通过 import 分析，检查是否存在以下违规方向：
  - `engine/` import `host/`、`services/`、`cli/`、`gui/`、`web/`、`wechat/`
  - `host/` import `services/`、`cli/`、`gui/`、`web/`、`wechat/`
  - `services/` import `cli/`、`gui/`、`web/`、`wechat/`
- 同时检查 `startup/` 是否 import 了 `services/`、`host/`、`engine/`（startup 只做启动期装配，不应依赖请求期组件）
- 判定标准：任何 `from dayu.xxx import` 或 `import dayu.xxx` 出现违规方向即报告

**1.2 跨层穿透调用**
- 检查是否存在 UI 层直接调用 Host/Agent 层（跳过 Service）
- 检查是否存在 Service 直接调用 Agent 层（跳过 Host）
- 检查上层是否在做下层参数类型化/装配，比如UI成装配DocToolLimits
- 例外：startup preparation 可以创建 Host 实例供 UI 持有

**1.3 preparation 模块职责越界**
- `startup/`：是否包含业务逻辑或请求期控制流？（应只做启动期依赖收敛）
- `services/contract_preparation.py`：是否包含业务决策？（应只做机械装配）
- `host/scene_preparer.py`：是否包含业务语义理解？（应只按 scene 策略机械收敛）

**1.4 接口设计的下层视角检查**
- 从 Agent（`engine/`）开始反向检查：假设 Host/Service/UI 不存在，`engine/` 的 public 接口是否自洽、是否依赖了不该知道的上层概念？
- 同样检查 Host（`host/`）：假设 Service/UI 不存在，接口是否自洽？
- 关注点：参数类型是否引入了上层定义的类型、方法命名是否暗含上层语义

**1.4.1 公共真源与旧实现对象是否已切断直连**
- 对已经在 `contracts/`、`execution/`、`services/` public preparation 中建立稳定真源的概念，检查上层模块是否仍直接 import 旧实现对象并把它们当作公共契约使用。
- 重点检查：运行配置、执行快照、Execution Contract、tool config、scene/model 解析结果等是否仍被 UI / Service / 测试代码直接绑定到 `engine/` 或 `host/` 内部实现类。
- 特别关注“运行时因字段兼容或 duck typing 暂时能工作，但静态契约已经漂移”的情况；这种情况也应报告，不视为合理兼容。
- 判定标准：若上层绕过稳定真源，直接依赖下层旧实现类、旧 dataclass、旧宽对象或历史兼容字段推进主链路，即报告。

**1.4.2 内部实现对象的暴露面是否继续诱导上层误用**
- 对仅应留在本层内部或只应由最后一跳 adapter / builder 使用的实现对象，检查它们是否仍通过 `__init__.py`、public 模块导出、文档示例或便利 import 暴露给上层。
- 重点检查 `engine/__init__.py`、`host/` public 边界、以及 README / 示例代码是否仍把内部实现类呈现为“推荐直接 import 的公共接口”。
- 若某对象的合理使用范围已经收缩到“仅本层内部 + 最后一跳装配转换”，但仓库公开面仍把它暴露成通用 API，应标记为架构收口候选。
- 判定标准：若内部实现对象的公开暴露面会诱导 UI / Service / 测试继续直接依赖，导致真源漂移难以彻底收口，即报告。

**1.5 职责混淆**
- 同一模块内是否混合了数据处理、持久化存储、外部调用三种以上职责
- 判定标准：单个类同时持有 I/O handle + 业务计算 + 对外 API 调用 → 报告

**1.6 模块/类膨胀**
- 单文件超过 600 行（不含空行和注释） → 标记为候选拆分项
- 单类超过 15 个 public 方法 → 标记为候选拆分项
- 非阈值硬限，但需要给出判断：膨胀是因为职责聚合合理还是应拆分

**1.7 不必要的聚合点**
- 是否存在某个函数/类作为多个独立功能分支的汇合点，对所有分支做了统一前处理/后处理，而这些分支实际应该各自独立处理
- 常见模式：一个 dispatch 函数既做参数转换又做路由又做结果合并

**1.8 各种反模式**
- 重点检查是否存在God object / God function / God dataclass / god bag / god builder 反模式：
  - **god-bag dataclass**：一个 dataclass / NamedTuple 把属于不同架构层的对象打包进同一个扁平容器。
  - **god-bag dataclass**：一个 dataclass / NamedTuple 把属于不同执行分支的对象打包进同一个扁平容器。
  - **跨层 builder 函数**：一个函数在内部穿越多个架构层完成创建和装配
- **越权编排其它层内部构件**：某一层的 builder 直接实例化另一层的内部实现类。
- 判定标准：函数或类的创建/持有对象跨越两个以上架构层
- 合法例外：UI 入口点（`cli/main.py` 等）作为组合根（composition root）按顺序调用各层的 public 工厂函数是正常的，但它不应该触碰任何层的内部实现类

**1.9 硬编码业务逻辑**
- 是否存在本该由通用语义定义区分的不同业务逻辑，由magic number / magic string判断决定。

### 2. OpenAI API 协议（仅 engine 部分）
- **2.1 请求消息结构是否符合协议**
  - 检查发往 OpenAI Chat Completions 的 `messages` 是否满足协议要求，尤其是 `role`、`content`、`name`、`tool_calls` 等字段的出现位置和类型。
  - 重点检查是否错误混入自定义字段、错误 role、错误 content 形状，或把上层内部语义直接塞进协议层消息结构。
  - 判定标准：请求 payload 中任一消息不符合 Chat Completions 规范 → 报告。

- **2.2 assistant tool_calls 消息是否构造正确**
  - 检查 assistant 触发工具调用时，是否正确使用 `tool_calls` 数组表达调用意图；`content` 为空、缺失或为文本时是否都按协议允许的方式处理。
  - 检查每个 tool call 的 `id`、`type`、`function.name`、`function.arguments` 是否完整且类型正确。
  - 判定标准：tool call 被编码成非协议字段、缺失必需字段、或参数承载方式与协议不符 → 报告。

- **2.3 tool 响应消息是否与 tool_call 严格对应**
  - 检查工具执行结果回填给模型时，是否使用 `role=tool`，并带上正确的 `tool_call_id` 与调用顺序。
  - 检查多 tool call 场景下，是否存在错配、串位、漏回填、重复回填。
  - 判定标准：tool message 与对应 assistant tool_call 不能一一对应，或顺序/标识不一致 → 报告。

- **2.4 流式增量解析是否符合协议语义**
  - 检查 streaming 模式下，对 `delta.role`、`delta.content`、`delta.tool_calls`、`function.arguments` 分片拼接的实现是否正确。
  - 重点检查多个 chunk、多个 tool call、arguments 分段追加时，是否会丢字、串 call、提前闭合或重复追加。
  - 判定标准：增量解析后的最终 assistant message 与协议语义不一致 → 报告。

- **2.5 finish_reason 是否被正确处理**
  - 检查 `stop`、`tool_calls`、`length`、`content_filter` 等终止原因是否被区分处理，而不是统一当作“正常完成”。
  - 判定标准：截断、内容过滤或需继续处理的响应被误判为正常终态，或 tool_calls 终止被错误当作最终文本回复 → 报告。

- **2.6 空内容 / 缺字段 / 非法返回的健壮性是否足够**
  - 检查 engine 是否能处理 `content=None`、空 `choices`、缺失字段、畸形 tool arguments、非 JSON 文本等协议边界情况。
  - 判定标准：对协议允许或现实中常见的边界返回没有防御，导致崩溃、静默错判或错误推进状态机 → 报告。

- **2.7 协议层与产品层语义是否分离**
  - 检查 engine 是否只做协议适配、消息编解码和流解析，不在 OpenAI 协议层夹带业务语义判断。
  - 判定标准：协议适配代码中混入场景业务判断、金融规则或上层产品语义 → 报告。

- **2.8 对不支持特性的处理是否明确**
  - 当前 **暂不支持 `function_call` feature**。
  - 检查代码是否错误兼容、半兼容或把 `function_call` 与 `tool_calls` 混为一谈。
  - 判定标准：出现对 `function_call` 的错误解析、错误回填或混用逻辑 → 报告；不要把“未支持 function_call”本身当作问题。

### 3. Agent 编写最佳实践（仅 engine 部分）
- **3.1 tool 调用循环是否有明确边界**
  - 检查 Agent 是否有清晰的停止条件：模型给出最终回复、无更多 tool_calls、达到最大迭代数、被取消、被宿主超时中止等。
  - 判定标准：tool loop 可能无界重试、无界自问自答，或缺少硬性停止条件 → 报告。

- **3.2 每轮状态推进是否单调且可解释**
  - 检查一次 agent iteration 内的状态推进是否清晰，例如：请求模型 → 解析回复 → 执行工具 → 回填结果 → 进入下一轮 / 结束。
  - 判定标准：同一轮中存在重复推进、跳步推进、失败后继续错误推进，或状态机分支难以证明正确 → 报告。

- **3.3 tool 调用是否合法且经过校验**
  - 检查 Agent 在执行 tool 前，是否校验 tool 名称存在、参数可解析、参数类型合理、调用是否重复。
  - 判定标准：对模型生成的 tool call 直接盲信执行，缺少最基本校验或去重保护 → 报告。

- **3.4 tool 结果回填是否完整且只回填一次**
  - 检查 tool 成功结果、失败结果、结构化错误是否都能稳定回填给模型或上抛，而不是静默丢失。
  - 检查同一 tool result 是否可能被重复注入上下文，或未注入却进入下一轮。
  - 判定标准：tool 执行事实与后续上下文不一致 → 报告。

- **3.5 tool 失败后的行为是否稳定**
  - 检查工具失败时，Agent 是显式把错误回给模型、转成终态错误、还是走有限重试；不要让失败隐式吞掉。
  - 判定标准：tool 失败后静默继续、无限重试同一失败调用、或把失败误当成功结果继续推理 → 报告。

- **3.6 对模型输出的不可信边界是否足够谨慎**
  - 检查 Agent 是否假定模型总会返回合法 JSON、合法 tool 名、合法参数、合法停止信号。
  - 判定标准：代码把模型输出当作可信输入直接驱动关键执行路径，缺少解析失败和非法输出防御 → 报告。

- **3.7 context window 管理是否保留最小继续执行事实**
  - 检查消息裁剪、摘要、token 预算控制是否保留 system 指令、最近用户请求、最近 tool 调用与关键结果，避免把继续执行所需事实裁掉。
  - 判定标准：裁剪策略只关注长度，不关注语义最小闭包，导致下一轮推理建立在不完整上下文上 → 报告。

- **3.8 重复调用与回环是否被抑制**
  - 检查 Agent 是否能识别明显重复的 tool call、同参回环、无信息增益的重复推理，并用 guard 或上限阻断。
  - 判定标准：模型一旦陷入重复工具调用，Agent 没有任何保护机制 → 报告。

- **3.9 取消、超时、中断信号是否被及时响应**
  - 检查 Agent 在模型等待、tool 执行前后、轮次切换处，是否及时检查取消或中断信号，而不是要等整轮跑完。
  - 判定标准：取消/超时信号到达后 Agent 仍长时间继续推进下一步执行 → 报告。

- **3.10 可观测性是否足以支撑定位 root cause**
  - 检查 iteration、tool call、模型回复、错误、截断、终止原因是否有足够 trace / event / log 支撑定位问题。
  - 判定标准：出现执行异常时，无法从 engine 侧还原“哪一轮、哪个 tool、因为什么而结束/失败” → 报告。

### 4. 多轮会话 编写最佳实践
- **4.1 状态真源是否唯一**
  - 检查多轮会话相关状态（session、run、pending turn、message transcript、resume snapshot）的真源是否唯一。
  - 重点看是否存在两个以上可写状态源同时描述“当前轮进行到哪里”。
  - 判定标准：同一事实若可由 UI 内存、Service 缓存、Host 存储、Agent 上下文中两个或以上位置独立修改 → 报告。

- **4.2 会话状态归属是否正确**
  - 检查 UI、Service、Host、Agent 各自是否只持有本层应持有的会话信息。
  - UI 只应持有展示态和用户交互态；Service 只应持有业务受理语义；Host 应拥有执行态、生命周期与恢复态；Agent 只应接收本次执行所需输入。
  - 判定标准：UI/Service 持有或决定可恢复执行状态，或 Agent 反向决定会话治理策略 → 报告。

- **4.3 恢复 / 重试语义是否稳定**
  - 检查 resume / retry / redelivery 是否基于持久化快照或宿主记录的稳定事实，而不是从当前 UI 输入、当前配置或 transcript 临时反推。
  - 检查恢复后是否会错误应用新的 override、prompt、config 或运行参数，导致同一轮语义漂移。
  - 判定标准：恢复逻辑依赖“当前环境猜测过去状态”，而不是依赖当时已接受的执行事实 → 报告。

- **4.4 单轮边界是否清晰**
  - 检查 conversation turn、agent iteration、tool batch、reply delivery 的边界是否清晰，是否混用了 turn / iteration / run 等概念。
  - 重点看 message 追加、tool result 回填、assistant reply 提交是否存在跨轮串写、重复提交或提前提交。
  - 判定标准：一个标识符同时承担两个以上生命周期概念，或提交时点早于宿主确认边界 → 报告。

- **4.5 并发与重入是否受宿主管理**
  - 检查同一 session 下并发发言、重复点击发送、外部重试、进程重启后的恢复，是否由 Host 统一治理。
  - 重点看是否存在 UI 侧布尔标记、Service 侧临时锁、Agent 内部状态来承担本应由 Host 负责的并发治理。
  - 判定标准：并发控制真源不在 Host，或不同入口对同一 session 采用不同并发规则 → 报告。

- **4.6 失败、取消、超时后的会话一致性**
  - 检查 tool 失败、模型失败、用户取消、超时、中断重启后，会话是否保持一致，不出现“用户看见已完成但系统不可恢复”或“系统继续运行但用户侧已结束”的分裂状态。
  - 检查 reply 是否只在可确认时落库 / 挂起 / 标记待投递，避免半提交。
  - 判定标准：终态与可恢复态不一致，或失败后 transcript / pending turn / run state 之间出现逻辑冲突 → 报告。

- **4.7 多入口行为是否一致**
  - 检查 CLI、Web、WeChat 等入口对多轮会话是否复用同一 Service 契约与 Host 能力，而不是各自实现一套 resume、pending、session 管理。
  - 判定标准：不同 UI 入口对同一会话语义有分叉实现，导致恢复、取消、历史回放行为不一致 → 报告。

- **4.8 上下文构造是否服务于多轮稳定性**
  - 检查历史消息裁剪、摘要、system prompt 注入、tool trace 回填是否保持多轮语义连续，不丢失继续执行所需的最小信息。
  - 判定标准：上下文裁剪后导致宿主记录仍声称“可继续”，但 Agent 实际已失去继续所需事实 → 报告。

### 5. “宿主强约束下的 LLM in the loop” 编写最佳实践
- **5.1 宿主是否真正拥有执行生命周期**
  - 检查 run 的创建、开始、取消、超时、完成、失败、恢复是否由 Host 显式拥有，而不是散落在 UI、Service 或 Agent 中。
  - 判定标准：存在非 Host 模块直接决定 run 生命周期推进、终止或恢复 → 报告。

- **5.2 资源边界是否由宿主收口**
  - 检查 timeout、并发上限、重试预算、tool 调用上限、token 预算、工作目录、网络/文件访问边界等是否由 Host 或 Host public scene preparation 统一落实。
  - 判定标准：资源治理规则散落在 UI / Service / Agent 自行判断，或同一限制存在多个不一致实现 → 报告。

- **5.3 Host状态机是否正确**
  - 检查turn和outbox的：创建、开始、取消、超时、完成、失败、恢复、孤儿 ，状态转换是否正确。
  - 特别检查异常情况下状态机处理。

- **5.4 Agent 是否退化为“消息交互执行器”**
  - 检查 Agent 是否只负责在给定输入和约束内进行消息交互、tool 调用编排与结果返回。
  - 重点检查 Agent 是否混入业务受理、执行策略选择、宿主治理、会话管理、恢复策略判断。
  - 判定标准：Agent 代码显式理解业务流程、决定是否接受请求、决定运行时治理规则 → 报告。

- **5.5 Service 是否越过 Host 直接控制 Agent**
  - 检查 Service 是否通过 Host public contract 发起执行，而不是直接拼装 Agent 输入、直接调用 engine runner、直接操纵 tool loop。
  - 判定标准：Service 绕过 Host 直连 Agent / engine，或直接依赖 Host 内部构件完成执行编排 → 报告。

- **5.6 Host 能力是否对执行形态保持通用**
  - 检查 Host 的治理能力是否同时适用于普通 Agent、金融专门 Agent、direct operation、复合流水线，而不是把某一具体业务路径硬编码进 Host。
  - 判定标准：Host 中出现仅服务某单一业务场景的专用业务语义分支，导致治理能力无法复用 → 报告。

- **5.7 scene preparation 是否保持机械收敛**
  - 检查 scene preparation / prompting 是否只根据 scene 定义、执行选项和静态策略机械地产生 Agent Input。
  - 判定标准：scene preparation 根据业务语义临场做“该不该查某类金融信息”“该走哪类研究策略”之类判断 → 报告。

- **5.8 失败处理是否由宿主兜底**
  - 检查 tool 失败、模型异常、取消、中断、投递失败后，最终失败语义、可恢复性、事件记录、外部可见状态是否由 Host 兜底收敛。
  - 判定标准：失败处理依赖 Agent 临场自救才能保持系统一致，或 UI/Service 自行补洞修正 Host 状态 → 报告。

- **5.9 执行契约是否稳定且分层清晰**
  - 检查 Service 输出给 Host 的是稳定 Execution Contract，Host 内部再机械收敛为 Agent Input；不要把两者混成一个跨层万能对象。
  - 判定标准：存在同时承载 UI、Service、Host、Agent 多层对象或语义的 god-bag / god-builder，用于“省事地把所有东西传下去” → 报告。

- **5.10 review 时的判断优先级**
  - 若发现执行问题，优先沿 `UI -> Service -> Contract preparation -> Host -> scene preparation -> Agent` 检查责任归属。
  - 只有在证据直接显示 Agent 本身错误时，才把 root cause 定位到 Agent；禁止因为现象出现在 Agent 输出上，就跳过 Host / Service 直接归因给模型或 prompt。

### 6. 金融逻辑分层
- 金融逻辑（SEC Item 编号解析、财务指标判断、filing 结构规则等）是否只出现在 `fins` 中。
- `engine` 中是否混入了金融逻辑。
- 金融逻辑中是否有规则硬编码（如公司名、固定 ticker）。
- 同一 Form 的主路由 processor 与 fallback processor 之间是否存在互相 import。
- 重点审查processor 提取数据的逻辑是否有bug。

### 7. 运行参数传递
- config参数是否最终传到Host / Agent。
- prompt asserts是否最终传到Agent。
- CLI 参数 是否作为override 最高优先级最终传到Host / Agent。

### 8. 代码质量
- 是否有硬编码 magic string / magic number（出现在生产路径中的）；工具 schema例外，工具 schema必须把字符串写在schema内，禁止使用常量定义。
- 调用函数，被调用函数的签名是否和调用方期望的一致。
- 函数是否使用object、any、json等无法进行严格类型检查的参数类型和返回值。
- 是否有死代码（未被调用的函数/类/分支）。
- 是否有明显 BUG（逻辑错误、边界条件遗漏、资源未释放等）。
- 是否有重复逻辑应抽取为公共函数。
- 是否有应该移除的兼容性代码：
  - 兼容性 re-export — import 了但本模块不引用，仅为保持旧导入路径。
  - 兼容性常量 re-export — 同上，针对常量。
  - 兼容性 wrapper/facade 方法 — 方法体仅透传到真源模块，不含额外逻辑。
- 是否pyright无错误。
- 是否符合 AGENTS.md 的代码结构要求（函数扁平化、无嵌套类、docstring 等）。

### 9. 性能（静态分析，无需 profiling）
仅标记通过代码阅读即可判断的明显性能问题，不做猜测性优化建议。关注以下模式：
- **循环内重复计算**：循环体内每次调用开销大的函数（如正则编译、文件 I/O、重复 JSON 解析），应提到循环外。
- **低效数据结构**：用 `list` 做成员查找（`x in list`）而应用 `set`；用字符串拼接替代 `join`。
- **不必要的全量加载**：一次性读取超大文件/列表到内存，而实际只需前 N 条或流式处理。
- **重复正则编译**：`re.compile` 在函数内每次调用时执行，应提取为模块级常量。
- **同步阻塞 I/O 在异步上下文中**：`async` 函数内调用同步阻塞（如 `requests`、`open` 大文件），应改用 `asyncio` 对应接口。
- **N+1 查询模式**：循环内逐条触发 I/O（文件读取、HTTP 请求），可批量合并。
- **重点审查processor 提取数据的逻辑是否有性能瓶颈**。
